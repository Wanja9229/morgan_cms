# 전투 시스템 (Battle System)

> 확산성 밀리언 아서 스타일 — 협동 레이드 전투
> 탐사 이벤트에서 몬스터/보스 발견 → 커뮤니티 전체 알림 → 참여자들이 기력를 소모해 공격

---

## 1. 개요

### 핵심 컨셉
- **비동기 협동 레이드**: 실시간 동기화 없이, 각자 기력가 있을 때 전투 페이지에서 행동
- **기력(氣力) 게이팅**: 전투 전용 자원 "기력"이 시간에 따라 충전 → 행동 빈도 자연 제한
- **탐사 연동**: 기존 탐사 시스템의 이벤트로 전투 조우 발생
- **참여 제한 없음**: 기력가 자연적 분산 장치 → 인원 제한 불필요
- **동시 다발 레이드**: 바쁜 시간대 3~5개 레이드 동시 진행 → 유저가 골라서 참여
- **커뮤니티 소통 기반 전략**: 유저들이 오너게시판/메모란에서 어떤 보스에 집중할지 상의
- **서버 권위**: 모든 계산은 서버에서 수행, 클라이언트는 결과 표시만

### 실시간 불필요 근거
| 요소 | 이유 |
|------|------|
| 기력 | 30분~1시간에 1 충전 → 분당 행동 수 극소 |
| 비동기 | 각자 여유 있을 때 행동, 동시 조작 불필요 |
| 반격 | 공격 시 즉시 반격 계산 → 턴 대기 없음 |
| 상태 갱신 | 전투 페이지 30초 폴링이면 충분 |
| 소통 | 게시판에서 전략 논의 → 각자 판단 후 행동 |

### 동시 다발 레이드 & 자원 배분

```
바쁜 시간대: 레이드 A, B, C, D, E 동시 진행
유저 30명 활동 중, 각자 기력 3~5 보유

→ 30명이 5개 레이드에 흩어짐
→ 기력 없는 유저는 어차피 행동 불가
→ "한정된 기력를 어떤 보스에 쓸까" = 핵심 전략

소통 예시 (오너게시판):
  "B보스 HP 70% 남았는데 시간 얼마 안 남음, 여기 집중합시다"
  "A보스는 HP 낮으니까 2~3명이면 충분할 듯"
  "C보스는 포기하고 A, B에 집중!"
```

### 자원 명칭 정리

| 자원 | 코드 | 용도 | 단위 |
|------|------|------|------|
| **노동력** | `stamina` (기존) | 개척 시설 건설 + 탐색 파견 | 유저 단위 |
| **기력** | `battle_energy` | 전투 행동 (공격/스킬/힐) | **캐릭터 단위** |

> 노동력과 기력은 완전히 독립. 전투가 개척 진행을 방해하지 않음.
> 노동력은 건축/파견을 하나의 풀에서 배분 (2종 분리 불필요 — 배분 자체가 전략).

### 디자인 원칙: 단순 시작, 필요 시 확장
- 탱커/서포터 역할이 밸런스상 안 맞으면 → 전원 딜러 체제로 전환 가능
- 스킬/장비 확장은 기본 시스템 검증 후 점진적으로

## 2. 스탯 시스템

### 2.1 기본 스탯 (4종)

| 스탯 | 코드 | 역할 | 스케일링 대상 |
|------|------|------|-------------|
| 체력 | `hp` | 생존 / 탱커 | 최대 HP. 많이 찍으면 오래 살아서 더 많이 행동 가능 |
| 근력 | `str` | 물리 딜러 | 일반공격(1소모) 데미지. 기력 효율 빌드 |
| 솜씨 | `dex` | 스킬 딜러 | 공격 스킬(2소모) 데미지. 한방 빌드 |
| 지력 | `int` | 서포터 | 힐량, 버프 효과, 디버프 효과 |

### 2.2 파생 수치 (스탯 + 장비에서 계산)

| 수치 | 공식 (예시) | 설명 |
|------|------------|------|
| 최대 HP | `base_hp + (hp * 10) + 장비HP` | 전투 중 HP 상한 |
| 물리 공격력 | `str * 2 + 무기ATK` | 일반공격/강타 기반 데미지 |
| 스킬 공격력 | `dex * 2 + 무기SATK` | 공격 스킬 기반 데미지 |
| 서포트 위력 | `int * 2` | 힐량, 버프/디버프 효과 기반 |
| 방어력 | `(hp + str) / 4 + 방어구DEF` | 피격 시 데미지 감소 |
| 크리티컬률 | 장비 옵션에서만 획득 | 기본 5% + 장비 보정 |
| 크리티컬 배율 | 150% 고정 + 장비 보정 | 크리티컬 시 데미지 배율 |
| 회피율 | 장비 옵션에서만 획득 | 기본 0% + 장비 보정 |

> 크리/회피는 스탯이 아닌 **장비 옵션**으로만 획득 → 장비 선택에 깊이 부여
> 구체적 수식은 밸런스 테스트 시 조정. 여기서는 방향만 제시.

### 2.3 빌드 다양성

| 빌드 | 주스탯 | 플레이스타일 |
|------|--------|------------|
| 물리 딜러 | STR (+HP) | 일반공격(1소모) 위주, 기력 효율 좋음 |
| 스킬 딜러 | DEX (+HP) | 공격 스킬(2소모) 위주, 한방 강함 |
| 탱커 | HP (+INT) | 도발로 반격 흡수, 여유 시 힐/버프 |
| 서포터 | INT (+HP) | 힐+버프+디버프로 팀 생존 지원 |
| 하이브리드 | 분산 배분 | 상황 대응 유연 |

> STR vs DEX의 핵심 트레이드오프: "적게 써서 꾸준히(1소모)" vs "많이 써서 세게(2소모)"
> 기력가 제한 자원이므로 이 선택이 실제 의미를 가짐

### 2.4 스탯 포인트 배분

- 캐릭터 생성 시 **초기 스탯 포인트** 지급 (관리자 설정, 기본 20)
- 4개 스탯에 자유 배분
- **스탯 초기화 아이템**: 상점에서 판매, 사용 시 배분 포인트 전부 회수 → 재배분 가능
- 추후: 레벨업/업적으로 추가 포인트 획득 확장 가능

### 2.5 TRPG 스탯 시스템과의 관계

기존 `mg_trpg_*` 테이블은 TRPG 시트(CoC, D&D 등) 용도로 유지.
전투 시스템은 **별도 `mg_battle_stat` 테이블** 사용 — 목적과 복잡도가 다름.

## 3. 장비 시스템

### 3.1 장비 슬롯 (3종)

| 슬롯 | 코드 | 주요 효과 |
|------|------|----------|
| 무기 | `weapon` | ATK(물리공격력), SATK(스킬공격력) 보정 |
| 방어구 | `armor` | DEF(방어력), HP 보정 |
| 장신구 | `accessory` | 특수 효과 (크리티컬+, 회피+, 기력 회복+ 등) |

> 크리/회피는 스탯이 아닌 장비에서만 획득 → 장비 선택의 의미 강화

### 3.2 장비 데이터 구조

기존 `mg_shop_item`의 `si_type`에 전투 장비 타입 추가:
- `battle_weapon` — 무기
- `battle_armor` — 방어구
- `battle_accessory` — 장신구
- `battle_consumable` — 전투 소모품 (회복약, 부활, 기력 충전 등)
- `battle_skill_book` — 스킬북 (스킬 해금)

`si_effect` JSON 예시:
```json
// 무기 — 물리 특화
{"slot": "weapon", "atk": 20, "satk": 5}

// 무기 — 스킬 특화
{"slot": "weapon", "atk": 5, "satk": 20}

// 무기 — 밸런스
{"slot": "weapon", "atk": 12, "satk": 12}

// 방어구
{"slot": "armor", "def": 10, "hp": 50}

// 장신구 — 크리 특화
{"slot": "accessory", "crit_rate": 8, "crit_mult": 20}

// 장신구 — 회피 특화
{"slot": "accessory", "evasion": 10}

// 장신구 — 서포터용
{"slot": "accessory", "support_power": 15}

// 소모품 — 부활 아이템
{"type": "revive", "hp_percent": 50}

// 소모품 — HP 회복
{"type": "heal", "hp_amount": 100}

// 소모품 — 기력 충전
{"type": "stamina", "amount": 3}
```

### 3.3 장비 획득 경로

| 경로 | 설명 |
|------|------|
| 상점 구매 | 포인트로 기본 장비 구매 (주요 경로) |
| 전투 드랍 | 보스/몬스터 처치 시 확률 드랍 |
| 탐사 보상 | 탐사 이벤트 보상으로 장비 획득 |
| 업적 보상 | 특정 업적 달성 시 고급 장비 지급 |
| 개척 순위 | 개척 공헌도 상위 보상 |
| 미션 당선 | 미션 당선작 보상 |
| 의뢰 보상 | 운영진 의뢰 완료 보상 |

> 모든 경로가 결국 `mg_inventory`에 INSERT → 장비 장착은 전투 세팅에서 수행

### 3.4 장비 등급 (선택적 확장)

| 등급 | 코드 | 색상 | 스탯 보정 |
|------|------|------|----------|
| 일반 | `common` | 회색 | x1.0 |
| 고급 | `uncommon` | 초록 | x1.3 |
| 희귀 | `rare` | 파랑 | x1.6 |
| 영웅 | `epic` | 보라 | x2.0 |
| 전설 | `legendary` | 주황 | x2.5 |

> 초기 구현 시 등급 없이 진행해도 무방. `si_effect`에 수치만 다르게 설정하면 됨.

## 4. 기력 시스템

### 4.1 기본 설정 (관리자 조정 가능)

| 항목 | 기본값 | 설명 |
|------|--------|------|
| 최대 기력 | 10 | 상한 |
| 충전 간격 | 1800초 (30분) | 1 기력 충전 시간 |
| 초기 지급 | 5 | 시스템 첫 사용 시 |

### 4.2 서버사이드 계산 (크론/데몬 불필요)

```
현재 기력 = min(
    저장값 + floor((now - last_charge_at) / 충전간격),
    최대치
)
```

- 행동 시 서버에서 계산 → 차감 → `last_charge_at` 갱신
- 클라이언트는 다음 충전까지 남은 시간만 표시 (카운트다운)

### 4.3 HP 자동회복 (lazy 계산)

행동 시점에 경과 시간을 기반으로 HP를 lazy 계산하여 회복:

```
행동 시:
  경과_충전수 = floor((now - last_hp_regen_at) / 충전간격)
  if (경과_충전수 > 0) {
      회복량 = max_hp * 회복률(%) * 경과_충전수
      current_hp = min(current_hp + 회복량, max_hp)
      last_hp_regen_at = last_hp_regen_at + (경과_충전수 * 충전간격)
  }
```

| 항목 | 기본값 | 설명 |
|------|--------|------|
| 자동회복률 | 5% | 충전 1회당 max_hp 대비 회복량 (관리자 설정) |
| 주기 | 기력 충전과 동일 (30분) | 행동 시 lazy 계산, 크론 불필요 |

> `last_hp_regen_at`은 `mg_battle_energy` 테이블에 저장.
> 행동하지 않으면 회복이 적용되지 않지만, 다음 행동 시 누적 회복됨.

**의도된 긴장 곡선:**
```
전투 초반: 기력가 병목 (HP 넉넉, 기력 기다림)
전투 중반: 양쪽 다 부족 (HP 깎이고, 기력도 아낌)
전투 후반: HP가 병목 (기력 있어도 체력 부족)
         → 포션 구매(포인트 소모) 또는 힐러 의존(소통) 필요
```

**HP 회복 3단계 (효율순):**

| 방식 | 효율 | 소모 | 역할 |
|------|------|------|------|
| 자동회복 | 최하 | 무료 (시간만) | 반격 1회 복구에 1~4시간 |
| 포션 | 중간 | 포인트 (상점 구매) | 즉시 회복, 경제 소모 유도 |
| 힐러 | 최상 | 힐러의 기력 | 소통 유도, 협동의 핵심 |

### 4.4 기력 소모

| 행동 | 소모량 |
|------|--------|
| 일반공격 | 1 |
| 스킬 (2소모 계열) | 2 |
| 고급 스킬 (전체공격, 전체힐, 수호) | 3 |

### 4.5 기력 충전 아이템

상점에서 `battle_consumable` 타입으로 판매. 사용 시 즉시 충전.

## 5. 스킬 시스템

### 5.1 기본 커맨드

| 커맨드 | 코드 | 소모 | 대상 | 기반 스탯 | 설명 |
|--------|------|------|------|----------|------|
| 일반공격 | `attack` | 1 | 적 1체 | STR | 기본 물리 공격 |
| 강타 | `power_attack` | 2 | 적 1체 | STR | 강화 물리 공격 (1.5배) |
| 스킬공격 | `skill_attack` | 2 | 적 1체 | DEX | 스킬 데미지 |
| 전체공격 | `aoe_attack` | 3 | 적 전체 | DEX | 전체 데미지 (0.6배, 다중 몬스터 전용) |
| 힐링 | `heal` | 2 | 아군 3명 선택 | INT | HP 회복 (1인당 INT*2) |
| 전체힐 | `aoe_heal` | 3 | 아군 전체 | INT | 전체 HP 회복 (1인당 0.4배) |
| 버프 | `buff` | 2 | 아군 N명 선택 | INT | 대상의 ATK 또는 SATK 일시 증가 |
| 디버프 | `debuff` | 2 | 적 1체 | INT | 대상의 ATK 또는 DEF 일시 감소 |
| 도발 | `taunt` | 2 | 자신 | — | 이후 N회 행동 동안 모든 반격이 나에게 집중 |
| 수호 | `guard` | 3 | 자신 | — | 도발 + 받는 데미지 30% 감소 |
| 부활 | `revive` | 3 | 아군 1명 (전사) | INT | 전사한 아군 1명 부활 (HP = INT * 3) |

> **힐링이 3명 타겟인 이유**: 1명 힐로는 10명 전원 커버에 기력 20 필요 (비현실적).
> 3명 힐이면 4회(기력 8)로 전원 커버 가능. 힐러도 남는 기력로 다른 행동 가능하지만,
> 힐만 전담해도 기여도 가중치가 높아 MVP 경쟁에 충분히 참여 가능.

> **부활 스킬**: 고급 스킬 (INT 15 이상 + 상점 구매 필요). 부활 후 HP는 시전자의 INT에 비례.
> **자가 부활 아이템** (상점 소모품)은 비싼 가격으로 판매 — 죽으면 안 되는 긴장감 유지.
> 힐러의 고유 가치를 높이는 핵심 스킬.

### 5.2 스킬 슬롯

- 각 캐릭터는 **스킬 슬롯 3개** 보유 (일반공격은 기본 장착, 슬롯 불필요)
- 사용 가능한 스킬 목록에서 선택하여 장착
- 스킬 해금: 기본 제공 + 상점 구매 + 전투 보상 + 업적

### 5.3 스킬 해금 & 확장 (추후)

| 방식 | 설명 |
|------|------|
| 기본 제공 | 강타, 스킬공격, 힐링은 모든 캐릭터 기본 보유 |
| 상점 구매 | 전체공격, 전체힐, 도발, 수호, 버프, 디버프, **부활** 등 고급 스킬 |
| 전투 보상 | 보스 처치 시 확률적 스킬북 드랍 |
| 업적 | 전투 업적 달성 시 해금 |

### 5.4 버프/디버프/도발 지속

- **디버프 (보스 대상)**: **시간 기반** — N분간 유지 (기본 5분)
  - 예: "5분간 받는 피해 +10%" / "5분간 DEF -15%"
  - 비동기 환경에 적합: 시각 기반이라 행동 수에 영향받지 않음
  - 중첩 불가 — 같은 타입 재사용 시 갱신 (남은 시간 리셋)
  - `be_debuffs` (JSON): `[{type, stat, value, expires_at}, ...]`
- **버프 (아군 대상)**: 대상 아군 N명 선택 (`ally_multi`, 기본 2명), 해당 유저의 **다음 2회 행동** 동안 유지
  - 아군 선택형이라 소통 유도: "기력 많이 남은 사람 누구? 버프 줄게"
  - 대상 유저의 행동만 카운트하므로 다른 사람 행동에 소모되지 않음
  - `buffs_active` (JSON): `[{type, stat, value, remaining}, ...]` — 대상 슬롯에 기록
- **도발/수호**: 해당 전투 내 **N회 행동** 동안 유지 (기본 5회, 큐 기반)
  - 도발자가 전사하면 즉시 해제
- 전투 종료 시 모든 버프/디버프/도발 초기화

## 6. 전투 조우 (인카운터)

### 6.1 발생 조건

기존 `mg_expedition_event`에 **전투 이벤트 타입** 추가:

```
ee_effect_type = 'battle_encounter'
ee_effect = {
    "encounter_template_id": 5,    // 몬스터 템플릿 참조
    "encounter_type": "boss"       // boss | mob_group | story_boss
}
```

탐사 완료 시 이벤트 판정 → `battle_encounter` 당첨 시:
1. `mg_battle_encounter` 레코드 생성
2. 발견자 자동 참여 (슬롯 1)
3. **전체 알림 발송** (`mg_notification` + 헤더 배너)

### 6.2 몬스터 템플릿

관리자가 미리 등록해둔 몬스터 정보:

| 필드 | 설명 |
|------|------|
| 이름 | 표시 이름 |
| 이미지 | 몬스터 일러스트 |
| 유형 | `boss` (1체, 대형) / `mob` (3~5체, 소형) / `story_boss` (스토리 보스, 영구 진행) |
| HP | 총 체력 (보스: 수천~수만, 몹: 수백, 스토리 보스: 수만~십만) |
| ATK | 공격력 (반격 데미지 기반) |
| DEF | 방어력 (받는 데미지 감소) |
| 제한 시간 | 첫 액션 이후 격퇴 제한 시간 (초, 스토리 보스는 라운드 단위) |
| 보상 포인트 | 처치 시 기본 포인트 |
| 보상 드랍 | 드랍 테이블 (아이템 ID + 확률) |

### 6.3 다중 몬스터 (mob_group)

- 3~5체가 한 인카운터에 등장
- 각 몬스터는 개별 HP 보유
- 플레이어가 **타겟 선택** 후 행동
- 전체공격(`aoe_attack`) 사용 시 모든 생존 몬스터에 데미지
- 모든 몬스터 처치 시 인카운터 완료

### 6.4 참여 시스템

**참여 인원 무제한** — 기력가 자연적 분산 장치 역할:
- 동시 다발 레이드에서 유저가 기력를 나눠 써야 함
- 한 보스에 몰려도 기력 총량이 제한되어 순간 DPS 폭주 없음
- "자리 없음" 불만 제거 → 밀리언 아서처럼 누구나 때릴 수 있는 구조

```
참여 구조:
├── 발견자: 자동 참여 + 우선 알림 발송 권한
├── 초대: 발견자가 관계 캐릭터 1~2명에게 개인 알림 (자리 예약 아님)
└── 자유 참여: 전체 알림 보고 누구나 참여 가능
```

**초대 = 우선 알림:**
1. 발견자에게 "알릴 캐릭터 선택" UI 표시
2. `mg_relation`에서 발견자 캐릭터와 `active` 관계인 캐릭터 목록 조회
3. 최대 2명 선택 → 해당 유저에게 개인 알림 ("같이 하자!")
4. 슬롯 예약 아님, 빨리 와서 함께 전투하자는 의미

**자유 참여:**
1. 전체 알림 또는 전투 목록에서 레이드 확인
2. 전투 페이지 접속 → 캐릭터 선택 → "참여하기" 버튼
3. 참여 즉시 전투 가능 (입장 시 max_hp로 시작)

**참여 제한:**
- 한 유저는 한 인카운터에 **캐릭터 1개만** 참여 가능
- 다른 인카운터에는 다른 캐릭터로 참여 가능
- 기력은 캐릭터 단위이므로 캐릭터 A의 기력을 다 쓰고 캐릭터 B로 다른 전투 참여 가능
- DB 제약: `mg_battle_slot`에 `UNIQUE(be_id, mb_id)` 추가

**전투 목록 페이지의 역할:**
- 진행 중인 모든 레이드를 한 눈에 표시
- 보스별: 이름, HP바, 남은 시간, 참여자 수
- 유저가 "어디에 기력를 쓸지" 판단하는 핵심 UI

## 7. 전투 진행

### 7.1 전투 상태 흐름

```
discovered (발견됨, 대기 중)
  → active (발견자 첫 액션 시 → 타이머 시작)
  → cleared (모든 몬스터 처치 → 보상 분배)
  → failed (제한시간 초과 → 몬스터 도주 처리)
  → expired (발견 후 일정시간 내 첫 액션 없음 → 자동 소멸)
```

### 7.2 행동 처리 (핵심 로직)

**요청 → 서버 처리 → 응답** 1회의 HTTP 요청:

```
1. 트랜잭션 시작
2. 유효성 검증
   - 전투 상태 == active (또는 discovered → 첫 액션이면 active로 전환)?
   - 참여자인가?
   - 기력 >= 필요량?
   - 내 HP > 0? (전사 상태 아닌가?)
   - 대상 몬스터 HP > 0? (적 대상 스킬 시)
   - 대상 아군 존재? (아군 대상 스킬 시)
3. 기력 서버사이드 재계산 + 차감
4. 데미지/효과 계산
   - 일반공격/강타: ATK = str * 2 + 무기ATK - 몬스터DEF
   - 스킬공격: SATK = dex * 2 + 무기SATK - 몬스터DEF
   - 힐링: HEAL = int * 2 * 배율
   - 버프/디버프: 효과량 = int 기반
   - 도발/수호: 스탯 무관 (상태 적용)
   - 크리티컬 판정: rand(0,100) < crit_rate(장비) → 배율 적용
   - 활성 버프/디버프 보정 적용
5. 효과 적용
   - 공격: 몬스터 HP 차감 (SELECT ... FOR UPDATE → UPDATE)
   - 힐: 대상 아군 HP 회복 (max_hp 초과 불가)
   - 버프/디버프: 대상에 효과 기록
   - 도발/수호: 인카운터에 도발 상태 기록
6. 몬스터 반격
   ┌─ 도발 큐 비어있지 않음? → 반격 대상 = 큐[0].ch_id
   └─ 도발 큐 비어있음       → 반격 대상 = 행동자
   - 큐[0].is_guard == true면: 반격 데미지 * 0.7
   - 반격 데미지 = 몬스터ATK * (0.8~1.2 랜덤) - 대상 방어력
   - 회피 판정: rand(0,100) < evasion_rate(장비) → 회피 성공
   - 대상 HP 차감
   - HP <= 0 → 전사 처리 → 큐에서 해당 ch_id 전부 제거
   - 큐[0].remaining -1 → 0이면 큐에서 shift()
7. 행동 로그 기록 (mg_battle_log)
8. 몬스터 사망 체크
   - 해당 몬스터 HP <= 0 → 사망 처리
   - 모든 몬스터 사망 → 인카운터 cleared → 보상 분배
9. 트랜잭션 커밋
10. JSON 응답 반환
```

### 7.3 반격 메커니즘

- 유저가 행동할 때마다 몬스터가 반격
- **도발 미적용 시**: 행동한 유저에게 반격
- **도발 적용 시**: 도발자에게 반격 (행동자 보호)
- 반격 데미지 = `monster_atk * (0.8~1.2 랜덤) - player_def`
- 최소 데미지 보장: 1 (방어력이 아무리 높아도)
- 수호 스킬 적용 중이면 반격 데미지 30% 감소

### 7.4 도발 시스템 (큐 기반)

도발은 **큐(대기열)** 구조로 관리. 여러 탱커가 도발을 걸면 순차적으로 소모.

```
도발 큐 동작 원리:

  탱커A: 도발 3회 → 큐: [A:3]
  탱커B: 도발 3회 → 큐: [A:3, B:3]
  탱커C: 수호 5회 → 큐: [A:3, B:3, C:5(수호)]

  행동 1~3: 반격 → 탱커A에게 (A:3→2→1→0, 소진 → 큐에서 제거)
  행동 4~6: 반격 → 탱커B에게 (B:3→2→1→0, 소진 → 큐에서 제거)
  행동 7~11: 반격 → 탱커C에게 (C:5→4→3→2→1→0, 수호 감소 적용)
  행동 12~: 도발 없음 → 행동자에게 반격
```

**큐 규칙:**
- 도발/수호 사용 시 큐 **맨 뒤에** 추가 (FIFO)
- 맨 앞 탱커의 잔여 횟수가 0이 되면 큐에서 제거 → 다음 탱커 활성화
- 탱커가 전사하면 해당 항목 즉시 큐에서 제거 → 다음 탱커 활성화
- 같은 탱커가 연속 도발 시 기존 항목에 횟수 **누적하지 않고** 새 항목으로 큐에 추가
  - (큐에 자기 차례가 다시 올 때까지 대기 → 다른 탱커와 교대 유도)
- 큐가 비어있으면 행동자에게 반격 (기존 방식)

**레이스 컨디션 방지:**
- `mg_battle_encounter.taunt_queue` (JSON 배열)에 큐 저장
- 행동 트랜잭션 내에서 `SELECT ... FOR UPDATE` → 큐 소모 → UPDATE
- 동시 요청 시 트랜잭션 직렬화로 순서 보장

```
커뮤니티 연동 활용 예시:
  탱커A (오너게시판): "도발 3턴 걸었습니다, 지금 공격해주세요!"
  탱커B: "A 끝나면 제가 이어갑니다, 도발 예약 걸어둠"
  딜러유저: "스킬 2발 넣겠습니다"
  서포터유저: "탱커A님 HP 반 이하인데 힐 넣을게요"
```

### 7.5 전사 & 부활

- HP가 0 이하 → **전사 상태** (`slot_status = 'dead'`)
- 전사 시 해당 전투에서 더 이상 행동 불가
- **부활 경로 2가지:**
  - **힐러 부활 스킬** (`revive`): 기력 3 소모, 전사 아군 1명 부활. 부활 HP = 시전자 INT * 3
    - 해금 조건: INT 15 이상 + 상점 구매 (고급 스킬)
    - 힐러의 고유 역할 가치 — 부활이 필요한 순간 힐러 의존도 상승
  - **자가 부활 아이템** (상점 소모품): HP를 `max_hp * 50%`로 회복
    - **비싼 가격** 설정 (포인트 경제 소모 유도)
    - 힐러 없이 혼자 대비하려면 비용 감수
- 부활 후 생존으로 전투 종료 시: 정상 보상 수령

### 7.6 타이머

- **시작 조건**: 발견자의 첫 번째 행동 (`battle_started_at` 기록)
- **제한 시간**: 몬스터 템플릿별 설정 (예: 보스 6시간, 일반몹 2시간)
- **만료 처리**: 크론 또는 다음 행동 시 서버에서 체크
  - 시간 초과 → `status = 'failed'`
  - 참여자에게 "격퇴 실패" 알림
  - 보상 없음 (또는 위로 보상 소량 지급 — 관리자 설정)
- **미시작 만료**: 발견 후 일정 시간(예: 2시간) 내 첫 액션 없으면 자동 소멸

### 7.7 스토리 보스 (영구 진행형 레이드)

일반 레이드와 달리 **잡을 때까지 반복 도전**하는 장기 콘텐츠.

**핵심 차이점:**

| 항목 | 일반 레이드 | 스토리 보스 |
|------|-----------|-----------|
| 실패 시 | 몬스터 소멸 | HP 유지 (소폭 회복 후 재등장) |
| 등장 방식 | 탐사 랜덤 | 탐사 랜덤 (이전 인카운터의 HP 계승) |
| 제한 시간 | 라운드 없음 (단일 타이머) | **라운드제** (라운드 실패 → 철수 → 재등장 대기) |
| HP | 수천~수만 | 수만~십만 (커뮤니티 전체가 장기간 공략) |
| 보상 | 처치 시 1회 | 처치 시 대형 보상 + 라운드별 참여 보상(소량) |

**라운드 시스템:**
```
스토리 보스 HP: 80,000

라운드 1 (탐사에서 발견):
  - 제한 시간 6시간, 참여자들이 공격
  - 시간 초과 시 → 보스 철수 (HP 60,000 남음)
  - 참여자에게 라운드 참여 보상 (기본 보상의 10~20%)
  - 보스 HP 소폭 회복: 잔여 HP의 N% 회복 (기본 5%)
  - 회복 후 HP: 63,000

라운드 2 (다음 탐사에서 재등장):
  - HP 63,000에서 시작
  - 시간 초과 시 → 다시 철수 + 소폭 회복

라운드 N (최종):
  - HP 0 → 처치 성공!
  - 전 라운드 누적 참여자에게 최종 보상 분배
```

**HP 회복 설계 의도:**
- 회복률이 너무 높으면 → 영원히 못 잡음 (좌절감)
- 회복률이 0이면 → 라운드 수만 채우면 무조건 잡힘 (긴장감 없음)
- **잔여 HP의 5%** 정도면: 딜이 회복보다 확실히 많아야 진전됨, 하지만 꾸준히 하면 반드시 잡을 수 있음

**DB 처리:**
- `mg_battle_monster`에 `bm_type = 'story_boss'` 추가
- `mg_battle_encounter`에 `be_story_hp_carry` (int) — 이전 라운드에서 넘어온 HP
- 스토리 보스 인카운터 실패 시: `status = 'failed'`로 마감, **다음 탐사 발견 시 새 인카운터를 생성하되 `be_story_hp_carry`에 잔여 HP(+회복분) 기록**
- 몬스터 템플릿에 `bm_story_regen_pct` (int, 기본 5) — 라운드 간 회복률
- 처치 완료 시: 해당 `bm_id`의 스토리 보스는 더 이상 탐사에서 등장하지 않음 (또는 관리자가 재활성화)

**최종 보상 분배 (라운드 누적):**
- `be_story_group_id`가 같은 모든 인카운터의 `mg_battle_slot` 기여도를 합산
- 누적 기여도 기준으로 상위 3명 보너스 + 전원 기본 보상
- 라운드별 참여 보상은 별도 즉시 지급

**운영 가이드 — 스토리 보스 재조우:**
- 스토리 보스는 탐사 파견에서 랜덤 등장하므로, **재조우 전용 탐색 파견**을 운영에서 구성 권장:
  - 짧은 파견 시간 (30분~1시간) + 적은 노동력 소모
  - 해당 보스의 `battle_encounter` 이벤트 확률을 높게 설정 (50~80%)
  - 기간제로 해당 탐색 지역을 열어둠 (보스 처치까지)
- 이렇게 하면 유저들이 적은 비용으로 빠르게 다음 라운드를 시작할 수 있음
- 시스템적 자동화 없이 **운영 설계로 해결** — 관리자가 탐사 지역/이벤트를 유연하게 구성

**관리자 설정 (`mg_config` 확장):**

| 키 | 기본값 | 설명 |
|----|--------|------|
| `battle_story_regen_pct` | `5` | 스토리 보스 라운드 간 HP 회복률 (잔여 HP의 %) |
| `battle_story_round_reward_pct` | `15` | 라운드 참여 보상 (기본 보상의 %) |

## 8. 보상 시스템

### 8.1 보상 배분

**기본 보상은 참여자 전원 균등. 기여도 상위 3명에게 추가 보너스.**

| 보상 | 기준 | 비고 |
|------|------|------|
| 기본 포인트 | 참여자 전원 균등 | 몬스터 템플릿에 설정된 기본값 |
| 1등 보너스 | 기여도 1위 | 기본 포인트의 50% 추가 |
| 2등 보너스 | 기여도 2위 | 기본 포인트의 30% 추가 |
| 3등 보너스 | 기여도 3위 | 기본 포인트의 15% 추가 |
| 발견자 보너스 | 발견자 | 기본 포인트의 20% 추가 |
| 아이템 드랍 | 확률 기반, 기여도 비례 | 기여도 높을수록 드랍 확률 상승 |

> 차등 보상이 아닌 **추가 보상** 방식 — 참여만 해도 기본 보상 보장, 열심히 하면 보너스.
> 소수 정예가 독식하는 구조를 방지하면서도 기여 동기는 유지.

### 8.2 기여도 계산

```
기여도 = 총 딜량
       + (힐량 * 1.0)
       + (버프 사용 횟수 * 50)
       + (디버프 사용 횟수 * 50)
       + (도발 흡수 반격 횟수 * 80)

기여율 = 내 기여도 / 전체 기여도 합계
```

> **힐량 1:1 환산**: 힐러가 힐만 전담해도 딜러와 대등한 기여도를 얻도록 설계.
> 힐러는 3명 타겟 힐(2소모) × 5회 = 기력 10으로 최대 15명분 회복 가능.
> 총 힐량이 자연스럽게 쌓여 딜러의 총 딜량과 비슷한 수준이 됨.
> 도발 흡수도 높은 가중치(80/회) → 탱커/힐러/딜러 모두 MVP 후보 가능.

**참여 판정 기준:**
- `mg_battle_slot.action_count > 0`이면 해당 전투의 **참여자**로 인정
- `action_count`는 **모든 행동**(공격, 힐, 버프, 디버프, 도발, 부활, 아이템 사용)에서 증가
- 따라서 힐러가 공격을 1번도 안 하고 힐/버프/부활만 해도 `action_count`가 쌓여 참여 인정
- 슬롯에 참여만 하고 행동을 0회 한 유저는 보상 대상에서 제외 (무임승차 방지)

### 8.3 드랍 테이블

몬스터 템플릿에 JSON으로 정의:

```json
{
    "drops": [
        {"item_id": 101, "chance": 30, "count": 1},
        {"item_id": 102, "chance": 10, "count": 1},
        {"item_id": 103, "chance": 3,  "count": 1}
    ]
}
```

- 각 참여자별로 독립 판정 (한 명이 먹는다고 다른 사람 못 먹는 게 아님)
- `chance`에 기여율 보정 적용
- 장비의 드랍률 보너스 옵션이 있으면 추가 보정

### 8.4 전사 패널티

- 전사 상태로 전투 종료 시: 보상 포인트 **50% 감소**
- 부활 후 생존으로 종료 시: 정상 보상
- 관리자 설정으로 패널티 비율 조정 가능

### 10.3 아군 대상 스킬 UI 플로우

공격/디버프는 **적 대상**, 힐/버프/부활은 **아군 대상** — 대상 선택 UI가 다름.

**공격 계열 (적 대상):**
1. 스킬 버튼 클릭 (일반공격, 강타, 스킬공격, 전체공격, 디버프)
2. 다중 몬스터(`mob_group`)인 경우 → 타겟 선택 UI에서 몬스터 클릭
3. 단일 보스인 경우 → 타겟 자동 확정
4. 서버 전송: `{be_id, action, target_idx}`

**아군 대상 (힐/버프/부활):**
1. 스킬 버튼 클릭 → **참여자 목록이 선택 모드로 전환**
2. 참여자 HP바 옆에 체크박스 표시
   - **힐링**: HP < max_hp인 아군만 선택 가능 (최대 3명)
   - **버프**: 활성 상태(`active`) 아군만 선택 가능 (N명)
   - **부활**: 전사 상태(`dead`) 아군만 선택 가능 (1명)
3. 대상 선택 후 "확인" 버튼 → 서버 전송: `{be_id, action, targets: [ch_id, ...]}`
4. 서버에서 시전자 슬롯의 `action_count++`, `total_heal += 힐량` 등 기여도 갱신

**핵심: 모든 행동은 전투 페이지(`be_id`) 안에서 발생**
- 전투 페이지 입장 = `be_id` 확정
- 참여자 목록 = 해당 `be_id`의 `mg_battle_slot` 조회
- 행동 API = `be_id` + `action` + `targets` → 서버에서 해당 인카운터의 슬롯에 기여도 기록
- 따라서 힐러가 공격 0회, 힐만 해도 **어떤 보스 전투에서 힐했는지** 명확히 추적됨

## 15. 밸런스 노트

### 참고 수치 (초기 세팅)

| 항목 | 값 | 근거 |
|------|-----|------|
| 일반몹 HP | 500~1,500 | 10명이 평균 데미지 50씩 10~30회 공격 |
| 보스 HP | 5,000~20,000 | 10명이 수 시간에 걸쳐 공략 |
| 몬스터 ATK | 20~80 | 플레이어 HP(200~400) 대비 5~15% |
| 플레이어 평균 HP | 200~400 | stat_hp 5~15 + 장비 보정 |
| 평균 일반공격 데미지 | 30~80 | str 5~15 → ATK 10~30 + 무기 |
| 스킬 데미지 | 50~150 | 일반공격의 1.5~2배 |
| 힐량 | 50~120 | 반격 2~3회분 |
| 도발 탱커 HP | 400~700 | HP 몰빵 → 반격 10~15회 견딤 |
| 동시 레이드 수 | 3~5개 | 탐사 발생률 + 제한시간으로 조절 |
| 레이드당 평균 참여 | 5~15명 | 유저 30명 기준, 기력 분산 |
| 스토리 보스 HP | 50,000~100,000 | 3~7라운드에 걸쳐 공략 (라운드당 ~15,000 딜) |
| 스토리 보스 회복 | 잔여 HP의 5% | 라운드별 회복, 딜량 > 회복량이면 반드시 진전 |
| 스토리 보스 라운드 보상 | 기본의 15% | 실패해도 참여 보상 → 동기 유지 |

### STR vs DEX 밸런스 기준

```
STR 딜러: 일반공격(1소모) * 10회 = str*2*10 = 20str 만큼의 누적 데미지
DEX 딜러: 스킬공격(2소모) * 5회  = dex*2*1.5*5 = 15dex 만큼의 누적 데미지

→ 같은 기력 10을 쓸 때:
  STR 20포인트 = 누적 400
  DEX 20포인트 = 누적 300 (크리 장비로 보완)
→ STR이 효율적이지만, DEX는 크리 장비와 시너지 + 전체공격 가능
```

### 힐러 효율 분석

```
INT 20 힐러, 기력 10 전부 힐에 투입:
  힐링(3명, 2소모) × 5회 = 15명분 회복
  1인당 회복량 = int*2*1.0 = 40
  총 힐량 = 40 * 15 = 600 → 기여도 600

STR 20 딜러, 기력 10 전부 일반공격:
  일반공격(1소모) × 10회
  1회 데미지 = str*2 + 무기 - 몬스터DEF ≈ 50
  총 딜량 = 500 → 기여도 500

→ 힐 전담 힐러의 기여도가 딜러와 동등하거나 약간 높음
→ 힐러가 힐만 해도 보상에서 불이익 없음
```

> 실제 밸런스는 테스트 플레이 후 조정.
> `mg_config`로 주요 계수를 관리자 설정 가능하게 두면 코드 수정 없이 조정 가능.
> 탱커/서포터 역할이 밸런스에 안 맞으면 도발/힐 스킬을 비활성화하고 전원 딜러 체제 전환 가능.
